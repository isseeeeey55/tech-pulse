# 2026年4月24日 New Relic アップデート解説

## はじめに

2026年4月24日付で、New Relic の重要なプラットフォームアップデート情報が公開されました。今回のアップデートは2026年3月に実施されたものを含み、主に**New Relic Lens**という新機能の導入が中心となっています。

注目すべきは、Snowflake、Google Sheets、各種データベースといった外部データソースへの直接クエリ実行が可能になった点です。これにより、従来は個別に管理されていた監視データと業務データの統合分析が実現し、SREの日常業務における根本原因分析の精度とスピードが大幅に向上することが期待されます。また、AI機能の強化により、複雑化するマイクロサービス環境での問題解決がより迅速になります。

## 注目アップデート深掘り

### New Relic Lens による外部データソース統合の実現

New Relic Lens は、オブザーバビリティの概念を大きく拡張する機能です。従来、New Relic で分析できるデータは、APMエージェント、Infrastructure Agent、ログフォワーダーなどを通じて収集されたテレメトリデータに限定されていました。しかし、実際のインシデント対応では、監視データだけでなく、業務KPI、顧客データ、コスト情報など、外部システムに保存されている情報との照合が不可欠です。

New Relic Lens により、Snowflake、Google Sheets、PostgreSQL、MySQLなどの外部データソースに対して、New Relic のコンソールから直接クエリを実行できるようになりました。これは単なるデータ取り込みではなく、**リアルタイムでの統合分析**を意味します。例えば、EC2インスタンスのCPU使用率が急増した際、同時にSnowflakeに格納されている売上データや注文件数の推移を確認し、ビジネスイベントとの相関を即座に把握できます。

**Before:**
```sql
-- New Relic でインフラメトリクスを確認
SELECT average(cpuPercent) FROM SystemSample 
WHERE hostname = 'prod-web-01' SINCE 1 hour ago

-- その後、別途 Snowflake にログインしてビジネスメトリクスを確認
-- 手動で2つのデータを突合
```

**After:**
```sql
-- New Relic コンソール上で統合クエリ（イメージ）
SELECT nr.cpuPercent, sf.order_count 
FROM SystemSample nr 
JOIN Snowflake.sales_data sf 
WHERE nr.timestamp = sf.timestamp
SINCE 1 hour ago
```

> **Note:** NRQL（New Relic Query Language）は、New Relicのデータベースに対するクエリ言語です。SQL風の構文でメトリクス、イベント、ログデータを分析できます。

この機能により、インシデント発生時の「データの切り替え作業」が不要になり、MTTR（平均修復時間）の大幅な短縮が期待できます。特にオンコール対応中の深夜帯では、複数システムへのログインや認証情報の管理負担が軽減され、初動対応のスピードが向上します。

### AI機能強化による異常検知と推奨アクションの自動提示

複雑化するマイクロサービスアーキテクチャでは、単一メトリクスの閾値監視だけでは問題の本質を捉えきれません。今回のアップデートで強化されたAI機能は、複数のシグナル（メトリクス、ログ、トレース）を横断的に分析し、異常パターンを自動検出します。

具体的には、Lambda関数の実行時間が通常の2倍に増加した場合、AIは関連するDynamoDBのレイテンシ、VPCフローログ、同時実行数などを自動的に分析し、「DynamoDBのプロビジョニングキャパシティ不足の可能性が高い」といった推奨アクションを提示します。これにより、経験の浅いエンジニアでも、ベテランSREと同等の初期判断が可能になります。

実際の運用では、アラート通知と共にAIの分析結果がSlackに投稿されるよう設定することで、オンコール担当者は現象の把握と初動対応を数分で完了できます。特にAWS環境では、サービス間の依存関係が複雑になりがちですが、Distributed Tracingと組み合わせることで、ボトルネックとなっているサービスを迅速に特定できます。

> **Note:** Distributed Tracing は、マイクロサービス間のリクエストフローを追跡し、各サービスの処理時間や依存関係を可視化する機能です。

## SRE視点での活用ポイント

### 監視・アラート・ダッシュボードの改善

New Relic Lens を活用することで、従来のインフラメトリクスだけでなく、ビジネスメトリクスも含めた複合ダッシュボードの構築が可能になります。例えば、「AWSコスト×リソース使用率×エラー率」を1つのダッシュボードに集約し、コスト効率の悪いリソースを視覚的に特定できます。

アラート設定では、外部データソースのKPIを条件に含めることができるため、「売上が一定以上かつエラー率が閾値を超えた場合のみ通知」といった、ビジネスインパクトを考慮したアラートが実現します。これにより、アラート疲れを軽減しつつ、本当に対応すべきインシデントに集中できます。

### AWS環境での実践的な運用Tips

**EC2環境:**
Infrastructure Agent を通じて収集したメトリクスと、CloudWatch Logs から転送したアプリケーションログ、さらにSnowflakeの業務データを統合分析することで、リソース使用量とビジネス負荷の相関を可視化できます。

**ECS/Fargate環境:**
コンテナメトリクスとGoogle Sheetsで管理しているデプロイスケジュールを照合し、デプロイ起因のパフォーマンス劣化を自動検出するダッシュボードを構築できます。

**Lambda環境:**
関数の実行メトリクスと外部APIレスポンスデータ（PostgreSQLに格納）を組み合わせ、サードパーティ依存による遅延を特定しやすくなります。

### アップグレード時の注意点

New Relic Lens は主にプラットフォーム側の機能追加であり、既存のエージェントアップグレードは基本的に不要です。ただし、最新のAI機能を最大限活用するには、Infrastructure Agent と APM エージェントを最新版に保つことを推奨します。

```bash
# Infrastructure Agent のバージョン確認
$ sudo newrelic-infra --version

# APT経由でのアップデート（Debian/Ubuntu）
$ sudo apt-get update
$ sudo apt-get install newrelic-infra

# YUM経由でのアップデート（RHEL/CentOS）
$ sudo yum update newrelic-infra
```

外部データソースへの接続設定では、認証情報の管理に注意が必要です。AWS Secrets Manager や Parameter Store を活用し、クレデンシャルをハードコードしない運用を徹底しましょう。

## まとめ

今回のアップデートは、New Relic が単なる「監視ツール」から「統合分析プラットフォーム」へと進化していることを示しています。特に New Relic Lens による外部データソース統合は、SREの業務フローを根本的に変える可能性を秘めています。

従来、インシデント対応では複数のダッシュボードやツールを切り替えながら情報を収集していましたが、これからは単一のコンソールで完結できます。AI機能の強化により、問題の早期発見と迅速な対応が可能になり、結果としてサービスの可用性向上とオンコール負担の軽減が実現します。

AWS環境を運用するSREにとって、Infrastructure AgentやAPMエージェントで収集したテレメトリデータと、ビジネスデータやコスト情報を組み合わせた分析は、今後の容量計画やコスト最適化において不可欠なアプローチとなるでしょう。まずは小規模な検証環境で外部データソース連携を試し、効果を実感してから本番展開することをお勧めします。
