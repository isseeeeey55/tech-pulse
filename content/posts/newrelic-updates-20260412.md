## はじめに

2026年4月12日のNew Relicアップデートは、計2件のリリースが発表されました。今回の注目ポイントは、セキュリティ強化を目的とした新機能と、Infrastructure Agentの重要なロールバック対応です。特に標準MFA機能の追加は、多くのSREチームが待ち望んでいた機能の一つとなるでしょう。一方、Infrastructure Agent 1.74.0については、緊急ロールバックが発生しており、運用への影響を慎重に検討する必要があります。

## 注目アップデート深掘り

### New Relic標準MFA機能の追加

New Relicが独自の多要素認証（MFA）機能を標準提供開始したことは、SREチームにとって重要なセキュリティ強化の選択肢が増えたことを意味します。従来、多要素認証を実装するには外部のSSO（Single Sign-On）サービスや、有料のアイデンティティプロバイダーとの連携が必要でした。今回の標準MFA機能により、これらの外部依存なしに、メールリンク認証という簡潔な方式でセキュリティレベルを向上できるようになりました。

SREの日常業務において、モニタリングツールへのアクセスセキュリティは極めて重要です。特に障害対応時の緊急アクセスや、複数の運用担当者が同じアカウントにアクセスする際の統制において、MFAは不可欠な要素となります。この新機能により、小規模から中規模のチームでも、追加コストなしでエンタープライズレベルのセキュリティを実装できるようになりました。

設定の有効化は、New Relicの管理画面から以下の手順で実行できます：

```bash
# New Relic CLIを使用した設定例（将来的な対応予定）
$ newrelic profile add --name production --api-key YOUR_API_KEY
$ newrelic auth configure --mfa-enabled true
```

Before状態では、パスワードのみでのログインが可能でしたが、After状態では登録メールアドレスに送信される認証リンクのクリックが必須となり、セキュリティが大幅に向上します。

### Infrastructure Agent 1.74.0の緊急ロールバック

Infrastructure Agent 1.74.0のBad releaseは、SREコミュニティにとって重要な教訓を含んでいます。このバージョンでは、最小権限サービスアカウントのサポート機能が一度実装されましたが、本番環境での安定性に問題が発見され、緊急にロールバックされました。この種の機能は、AWSのEC2インスタンスやECSタスクで動作するInfrastructure Agentにとって、セキュリティ強化の核心的な要素でした。

SREの運用において、エージェントのアップグレードは慎重な計画が必要です。特に今回のようなロールバックが発生した場合、現在のバージョンでの運用継続と、次期安定版までの待機戦略が重要になります。現在1.73.x系統で安定稼働している環境では、以下のようにバージョン固定を推奨します：

```bash
# パッケージマネージャーでバージョン固定する例
$ sudo apt-mark hold newrelic-infra
$ sudo yum versionlock newrelic-infra-1.73.*
```

また、Infrastructure Agentの設定ファイルで現在のサービスアカウント設定を確認し、権限の過不足がないかを再検証することが重要です：

```yaml
# /etc/newrelic-infra.yml の設定例
license_key: YOUR_LICENSE_KEY
display_name: production-server-01
verbose: 0
# サービスアカウント関連設定の確認
custom_attributes:
  environment: production
  service_account_validated: true
```

> **Note:** Infrastructure Agentは、システムメトリクス、プロセス、ネットワーク情報を収集してNew Relicに送信するエージェントソフトウェアです。

## SRE視点での活用ポイント

今回のアップデートは、セキュリティとリスク管理の両面でSRE業務に影響を与えます。標準MFA機能については、既存の監視ダッシュボードやアラート設定に影響を与えることなく、アクセスセキュリティのみを強化できる点が魅力的です。特にAWS環境でマルチアカウント構成を採用している組織では、各環境へのアクセス統制をNew Relic側でも強化できるメリットがあります。

すぐに試せるTipsとして、まず開発環境やステージング環境でMFA機能を有効化し、運用フローへの影響を検証することを推奨します。障害対応時の緊急アクセス手順も含めて、チーム全体での動作確認が重要です。

Infrastructure Agentについては、現在の安定版での運用継続を強く推奨します。特にECS FargateやLambda環境でInfrastructure Agentを活用している場合、予期しない動作変更は重大な影響を与える可能性があります。アップグレード計画は次期安定版のリリースを待ち、十分なテスト期間を設けて実施してください。

アップグレード時の注意点として、必ずカナリアデプロイメント方式を採用し、一部のインスタンスでの動作確認を経てから全体展開を行うことが重要です。また、ロールバック手順も事前に準備しておくことで、今回のような緊急事態にも迅速に対応できます。

今回のアップデートは、New Relicのセキュリティ機能の充実と、開発プロセスの透明性を示す内容となりました。標準MFA機能の追加は、多くのSREチームにとって待望の機能であり、セキュリティ強化の選択肢を大幅に広げるものです。一方、Infrastructure Agentのロールバック対応は、継続的なサービス改善における品質管理の重要性を改めて認識させてくれました。

SREとしては、新機能の積極的な活用と、安定性を重視した慎重なアップグレード戦略のバランスが求められます。特にクリティカルなプロダクション環境では、機能の魅力よりも安定性を優先し、十分な検証を経た導入計画を立てることが重要です。
